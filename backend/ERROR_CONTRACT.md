# Backend Error Contract

This project returns a consistent JSON error shape from the global error middleware.

## Success Response Shape

JSON success responses should follow this structure where possible:

```json
{
  "success": true,
  "message": "Optional success message",
  "...domainFields": {},
  "meta": {},
  "requestId": "uuid-or-request-id",
  "timestamp": "2026-03-26T00:00:00.000Z"
}
```

Use `sendSuccess(res, { statusCode, message, data, meta })` from `utils/successResponse.js`.
Legacy text endpoints (like ping) can remain plain text for compatibility.

## Response Shape

All API errors should follow this structure:

```json
{
  "success": false,
  "code": "ERROR_CODE",
  "message": "Human readable message",
  "details": {},
  "requestId": "uuid-or-request-id",
  "timestamp": "2026-03-26T00:00:00.000Z"
}
```

## Field Meaning

- `success`: Always `false` for errors.
- `code`: Stable machine-readable error code for UI logic and telemetry.
- `message`: User-safe message that can be shown in UI.
- `details`: Optional structured metadata (validation errors, conflicting fields, etc).
- `requestId`: Correlation ID for tracing logs and support/debugging.
- `timestamp`: UTC timestamp generated by the API.

## Recommended Status Code Mapping

- `400` invalid input, validation, bad state transitions
- `401` auth required, invalid/expired token
- `403` role/access denied
- `404` missing resource/route
- `409` conflict (duplicate/unique constraint when applicable)
- `429` rate limit exceeded
- `500` unexpected internal error
- `503` dependency unavailable (database/external services)

## Controller Guidelines

- Use `AppError` for expected business-rule errors:
  - `return next(new AppError('Message', 403, 'FORBIDDEN_CODE', { ...details }))`
- Prefer `errorFactory` helpers for common cases:
  - `validationError(...)`, `unauthorizedError(...)`, `forbiddenError(...)`, `notFoundError(...)`, `conflictError(...)`
- For unexpected failures, use `return next(error)`.
- Avoid direct `res.status(...).json(...)` for error responses in controllers.
- Wrap async route handlers with `asyncHandler(...)` in route files.

## Frontend Integration Notes

- Frontend should rely on `code` for deterministic behavior.
- `message` is safe to display to users.
- Include `requestId` in support logs/screenshots for issue triage.
